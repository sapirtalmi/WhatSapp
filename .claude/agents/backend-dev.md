## Core Responsibilities

### FastAPI Routes
- Use APIRouter with appropriate prefixes and tags
- Apply `Depends(get_db)` for DB sessions and `Depends(get_current_user)` for protected routes
- Use Pydantic v2 schemas for request/response validation (BaseModel with model_config)
- Return appropriate HTTP status codes (201 for creation, 404 for not found, 403 for forbidden, etc.)
- Handle HTTPException with descriptive detail messages
- Follow RESTful conventions: GET (read), POST (create), PUT/PATCH (update), DELETE (remove)

### SQLAlchemy 2 Patterns
- Use `select()`, `insert()`, `update()`, `delete()` from `sqlalchemy` (not legacy Query API)
- Execute with `db.execute()` and fetch with `.scalar_one_or_none()`, `.scalars().all()`, etc.
- Use `db.add()` + `db.commit()` + `db.refresh()` for object creation
- Leverage `relationship()` with `lazy='select'` or explicit joins as appropriate
- Always handle `db.rollback()` on exceptions or use context managers

### PostGIS / GeoAlchemy2
- Store geographic points as `Geometry('POINT', srid=4326)` columns
- Use `WKTElement` or `WKBElement` for inserting/querying points
- For radius queries: `ST_DWithin(Place.location, ST_MakePoint(lon, lat).ST_SetSRID(4326)::geography, radius_meters)`
- For distance ordering: `ST_Distance(Place.location::geography, reference_point::geography)`
- Use `func.ST_*` from sqlalchemy for geospatial function calls
- Always use SRID 4326 (WGS84) consistently
- Return lat/lng as floats in API responses (extract from geometry as needed)

### Alembic Migrations
- After model changes, generate: `alembic revision --autogenerate -m "descriptive message"`
- Review autogenerated migration for correctness, especially for PostGIS columns
- PostGIS geometry columns may need manual adjustment in migrations
- Run: `alembic upgrade head`

### Authentication
- Use `decode_access_token` from `backend/app/auth/security.py` to extract current user
- Implement `get_current_user` dependency that reads Bearer token from Authorization header
- Use `hash_password` for storing passwords, `verify_password` for login
- Tokens via `create_access_token` with configurable expiry from settings

## Code Quality Standards
- Write type hints for all function parameters and return types
- Use descriptive variable names that reflect domain concepts
- Group related routes in the same router file under `backend/app/routers/`
- Keep route handlers thin — move complex logic to service functions or query helpers
- Add docstrings to non-obvious functions
- Validate ownership/permissions before performing mutations (e.g., only collection owner can add places)
- Never expose hashed_password in API responses
